# AGENTS_KO.md

## 코딩 규칙

- 설정값, 파일 경로, 모델명, API URL, 인증 정보, 비즈니스 상수 및 레이블 텍스트 등의 데이터(한글 레이블 사전 정보 등 포함)는 코드 파일 내에 직접 작성(하드코딩)하여 남겨두지 않는다. 대신 기존 설정 파일(예: YAML), 환경 변수에서 로드하는 방식을 사용하여 데이터와 코드를 분리한다.
- 사소하거나 자명하지 않은 동작을 추가/변경할 때는 객체 지향 설계를 적용한다. 역할을 집중한 클래스를 사용하고 거대한 절차적 함수 작성을 피한다.
- 애플리케이션 로그 작성 시 처음부터 공통 로깅 클래스를 구축해야 하는 것은 아니나, 프로젝트의 전체 소스 코드 라인 수(줄 수)가 2000줄이 넘어가는 시점부터는 `common` 폴더 아래에 `logging`, `config`, `error` 처리 클래스를 별도로 생성하여 공용으로 이용하도록 일원화한다. 런타임 로깅을 위해 `print()`를 사용하지 않는다.

## 한국어 주석 작성 규칙

- 한글이 포함된 모든 파일은 UTF-8 인코딩으로 저장한다.
- 코드 주석과 docstring은 한글로 작성한다.
- 함수와 클래스에는 목적과 사용 의도를 설명하는 한글 docstring을 작성한다.
- 클래스 변수에는 값의 도메인 의미, 허용 범위, 이후 흐름에서 쓰이는 방식을 설명하는 한글 주석을 작성한다. 변수명이나 타입만 반복하는 주석은 금지한다.

## Python 파일 헤더 규칙

- 각 `.py` 파일 첫머리에는 작성일, 설계자, 설계자 소속, 설계자 이메일 주소를 주석으로 기록한다.
- 헤더는 module docstring, import, 실행 코드보다 먼저 배치한다.
- 헤더 형식은 다음 예시를 따른다.

```python
# 작성일: YYYY-MM-DD
# 설계자: 이름
# 설계자 소속: 회사명
# 설계자 이메일: name@example_corp.com, name@example_personnel.com
```

## README 및 설명 작성 규칙

- README 문서나 CLI 옵션 설명 작성 시, 'A는 A다' 식의 무의미한 동어반복 설명은 금지한다.
  - *잘못된 예: `--dialect: 사용할 dialect입니다.`*
  - *올바른 예: `--dialect: 쿼리를 실행할 대상 데이터베이스 문법 규격(예: duckdb, postgres)입니다.`*
- 사용자가 도구의 역할, 입력값의 도메인 의미, 그리고 설정의 영향 범위를 명확히 이해할 수 있도록 세부 용어 설명 및 구체적인 작동 방식과 예시를 포함하여 한글로 상세히 기술한다.
